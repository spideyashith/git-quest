## Step 4: Restructure Tree View for Hierarchical Display

**File:** `src/tree-view.ts`

### 4a. Add a tag-based approach to identify node types

No need for subclasses. Use the `description` or `id` property on `GitQuestTreeItem`, or simply store a `nodeType` field. Simplest approach: add a public `children` property to `GitQuestTreeItem` to store pre-built child items.

Alternatively, use a lightweight approach: tag each item with a `contextValue` or just match by label prefix in `getChildren`.

**Chosen approach:** Store child items directly on the parent `GitQuestTreeItem` via a `children?: GitQuestTreeItem[]` property. This avoids label-parsing and is clean.

```typescript
export class GitQuestTreeItem extends vscode.TreeItem {
  public children?: GitQuestTreeItem[];
  // ... existing constructor
}
```

### 4b. Rewrite `getChildren(element?)` 

**Root call** (`element` is `undefined`):
- Build the "Repository" node with `CollapsibleState.Expanded`
- Populate its `children` array:
  - `Current Branch: <branch>` (None)
  - `Modified Files (N)` (Collapsible) with children = file path items
  - `Staged Files (N)` (Collapsible) with children = file path items
  - `Repository Status: <status>` (None)
- Return `[repositoryNode]`

**Child call** (`element` is provided):
- Return `element.children || []`

### 4c. Build file path items with open command

For each modified/staged file, create a `GitQuestTreeItem` with:
- `label`: the file path (or just filename for readability)
- `collapsibleState`: `None`
- `command`: `{ command: 'vscode.open', title: 'Open File', arguments: [vscode.Uri.file(path)] }`

This uses VS Code's built-in `vscode.open` command to open the file in the editor.
